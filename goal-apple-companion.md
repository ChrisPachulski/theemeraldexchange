## Tier S0 — Production correctness (silent failures found live)

### 1. GlitchTip has captured zero real production errors since the relay was added — the configured DSN host is unresolvable from inside the backend's docker network
- **Where:** `server/services/serverTelemetry.ts:33-42`  (S · high value)
- **Failure:** Every server-side error the backend's onError handler tries to relay via serverTelemetry.ts silently fails (the fetch to the DSN host throws ENOTFOUND and is swallowed, per the module's own 'never fail a request over telemetry' design) — so GlitchTip has been blind to 100% of real production server errors since the relay was introduced (commit d6bfc70), creating false confidence that 'there are no errors' when the actual state is 'errors are generated but never delivered.'
- **Fix:** Point EEX_TELEMETRY_DSN at a host the backend's docker network can actually resolve (a LAN/reverse-proxy address, or join a Tailscale sidecar container to the same docker network as exchange-backend), not the bare Tailscale MagicDNS name. Add a startup self-check that DNS-resolves and probes the configured DSN host and logs loudly (not silently) on failure so this doesn't go unnoticed again.
- **Gate:** After repointing the DSN, trigger a deliberate server error (or the existing 'eex direct store test' self-check) and confirm a new row appears in exchange-glitchtip-db's issue_events_issue within the 2s relay timeout window — today that row never appears.
- **Verifier note:** Keep real=true, size S, value high as filed — but correct the mechanism text: drop the \"onError handler relays via serverTelemetry.ts\" claim (onError uses @sentry/node's own Sentry.captureException/transport with the same DSN; serverTelemetry.ts only carries background-job failures — recommender/iptv/token-sweep/db-backup) and drop the 401/503 examples as evidence (neither code path ever calls telemetry, so they don't illustrate the bug). Root cause and fix/gate are otherwise correct as filed.

### 2. cloudflared stale-netns outage recurs because the fix only runs inside a full `deploy-nas.sh` deploy, not on a standalone backend restart
- **Where:** `scripts/deploy-nas.sh:417-432 (force-recreate lives only in the deploy path); scripts/nas-canary-deploy.sh:48,165-168 (separate ytresolve canary also force-recreates, but only as a side effect of ITS OWN deploy)`  (S · high value)
- **Failure:** Any backend container recreation event that does NOT go through deploy-nas.sh's own force-recreate step (line 432) leaves the public API silently down until a human diagnoses it manually via `docker ps` uptime comparison — this has already happened at least once and nothing currently detects the drift automatically.
- **Fix:** Add a small cron/systemd-timer health probe on the NAS that runs every 1-2 minutes: compare `docker inspect -f '{{.State.StartedAt}}' exchange-backend` vs `exchange-cloudflared`; if backend's start time is newer than cloudflared's, run `docker compose up -d --no-deps --force-recreate cloudflared` automatically and log/alert. This is the exact 'watchdog' the incident doc and memory both call out as still missing.
- **Gate:** Manually `docker restart exchange-backend` (or kill -9 the backend PID) outside of any deploy script, then measure time-to-recovery of `curl -s -o /dev/null -w '%{http_code}' https://api.theemeraldexchange.com/api/health`: today this stays 530 indefinitely until a human intervenes (red); after the watchdog ships, it must self-heal to 200 within one watchdog interval (green) with no human action.

### 3. VOD / series / catch-up grant tokens use the 300s finite-asset TTL — the same bug that 'froze live cable after exactly 5 minutes', still unfixed for on-demand playback
- **Where:** `server/routes/iptv.ts:1080-1082 (vod), 1170-1172 (series), 738-743 (catchup); server/services/iptvHlsProxy.ts:104-110 (HLS segment tokens); context server/env.ts:683-693`  (S · high value)
- **Failure:** A user watches any movie or episode longer than ~5 minutes of buffered content. AVPlayer (progressive MP4) issues fresh HTTP range requests as playback advances or on any seek; hls.js (VOD .m3u8) fetches segment URLs across the whole runtime. Every such request past the 300s mark carries the now-expired grant/segment token → 401 invalid_token → playback stalls or ejects mid-film. Identical failure class to the live-cable freeze that was already fixed, but left in place for all on-demand content.
- **Fix:** Give VOD/series/catch-up grant tokens (and the HLS segment tokens minted from their manifests) a playback-duration TTL, mirroring MEDIA_STREAM_TOKEN_TTL_SECS=21_600 used for local media (env.ts:699). The tokens are rid-bound to one title+sub, so a long TTL is low-impact — exactly the rationale already accepted for local media and live.
- **Gate:** Test: mint a VOD grant token, advance a mock clock to 301s, replay a range GET on /stream/vod/:id/:ext — currently returns 401 (expired_token); assert 200 after the TTL fix. Red→green.

### 4. TMDB episode-metadata retry storm: no negative-result caching for permanently-unresolvable episodes
- **Where:** `crates/media-core/src/scanner.rs:639-657 (also crates/media-core/src/tmdb.rs:485-492)`  (M · high value)
- **Failure:** Every scheduled scan (~hourly) re-issues a live TMDB /tv/{id}/season/{s}/episode/{e} call for every episode file whose title is still NULL, because the only short-circuit is `title IS NOT NULL` (scanner.rs:639-657, and the identical pattern in backfill_metadata:724-742). For shows whose on-disk season/episode numbers don't line up with TMDB's numbering scheme (absolute-numbered anime, alternate-cut sitcoms, etc.) the 404 is permanent, so the same ~1,900+ episode rows across 15+ shows get re-probed forever: 44k wasted TMDB calls per 48h, other shows pushed into 429 backoff, and those episodes' titles/air-dates never appear in the library/guide UI for any user because there is no way for the code to mark an episode 'permanently unresolvable.'
- **Fix:** Add a `tmdb_lookup_failed_at`/attempt-count column on `episodes` and skip re-probing until a cooldown elapses (e.g. 30 days) or a max-attempt cap is hit; separately, use TMDB's `episode_groups` endpoint for shows flagged as absolute-numbered so file S/E can be mapped to the correct TMDB S/E instead of guessing 1:1.
- **Gate:** Run two consecutive scheduled scans and confirm `SELECT count(*) FROM episodes WHERE show_id=124 AND title IS NULL` stays at 1042 with zero new 'episode non-2xx for tv 111110' log lines on the second scan (currently it re-logs ~1010 every single scan).

### 6. recommenderCaller latches serverId failure for the entire process lifetime — a transient boot-time DB race permanently degrades suggestions, and GlitchTip is dead so nobody is alerted
- **Where:** `server/services/recommenderCaller.ts`  (S · medium value)
- **Failure:** If server.db is momentarily unavailable on the FIRST recommender call after a backend restart (volume-mount race at boot), the latch trips for the whole process. In the recommender's `enforce` mode every subsequent /api/suggestions request then 401s until a manual restart; in off/log mode users silently get generic TMDB trending instead of personalized recs. - **Fix:** Replace the permanent boolean latch with a bounded retry (e.g. re-attempt ensureServerId() after a short backoff, or every N minutes / next request past a cooldown) so a transient boot race self-heals; alternatively probe server.db readiness at boot before serving. Add a one-line telemetry emit on the demotion so it is observable.
- **Gate:** Unit test: stub ensureServerId to throw once then succeed; call getServerId() twice — assert the second call returns a non-null id (recovers). Red today (returns null forever after the first throw), green after the retry change. `_resetServerIdForTests()` already exists to isolate the latch.

## Tier S1 — IPTV resilience (root cause: 2026-07-06 Fox Soccer Plus incident)

### 7. Dead-feed placeholder (clean ffmpeg EOF) is never detected and never fails over to a sibling channel feed
- **Where:** `server/services/iptvLiveRemuxMap.ts:102-139 (classification) + server/services/iptvRemux.ts:367-371 (exit handler) + server/services/sourcePrecedence.ts:110-126 (buildCandidates)`  (L · high value)
- **Failure:** A viewer tunes a channel whose upstream feed is a dead-channel placeholder. ffmpeg copies the ~30s stub, EOFs cleanly (code 0), and the session dies. The backend re-dials the SAME dead stream_id (or serves a 503) instead of switching to any of the ~5 duplicate Fox Soccer Plus feeds carrying the same event. The user sees ~30s of a stub image (or a black screen) then a hard failure, on a channel a competing IPTV app would have recovered by switching feeds.
- **Fix:** In the remux exit path, capture exit code + wall-clock lifetime and tag a clean EOF under ~60s as a dead-feed. On that signal, have ensureLiveRemuxEntry consult a sibling-feed resolver: SELECT other channels sharing normalized name / epg_channel_id, and retry the next candidate feed before surfacing an error. Persist a short per-feed 'dead' memory (like the needsReencode TTL map) so the dead variant is skipped on the immediate retry.
- **Gate:** Unit test: stub startRemuxSession to spawn a process that exits code 0 after emitting 4 segments; assert ensureLiveRemuxEntry (given two channels with matching epg_channel_id) advances to the second stream_id rather than re-dialing the first. Red today (no sibling lookup exists), green after.

### 8. Remux manifest path returns a generic 503 'remux_warming' for an offline upstream — client cannot distinguish channel-offline from server-warming
- **Where:** `server/routes/iptv.ts:891-894 and 925-928`  (M · high value)
- **Failure:** When a channel's upstream feed is offline/dead, the Apple client receives a bare 503 that is byte-identical to the transient 'still warming up, retry in 3s' response. It cannot tell the user 'this channel is off the air upstream' vs. 'the server is busy', so it shows a generic retry/load-failure and keeps polling a dead feed instead of offering the user an alternative or a clear message.
- **Fix:** Split the two 503s: keep `remux_warming` (Retry-After) only for a genuinely still-starting session, and emit a distinct terminal code (e.g. `channel_offline_upstream`, non-retryable, matching the SourceUnavailablePayload shape) when a session died from a clean/too-fast EOF or exhausted its sibling-feed candidates. Add the code to the Swift Decodable closed enum in lockstep (contract §12.4).
- **Gate:** Integration test on the manifest route: simulate a session whose ffmpeg exited code 0 with no segments and assert the response carries a terminal `channel_offline_upstream` reason (not `remux_warming`). Red today; green after the split.

### 9. Concurrency cap (default 4) exceeds the upstream-connection cap (default 2): the 3rd+ concurrent remux viewer is granted, then silently ffmpeg-evicted with no client-visible reason
- **Where:** `server/services/iptvRemux.ts:230-242 (LRU eviction); server/env.ts:674 vs 682 (cap mismatch)`  (M · medium value)
- **Failure:** Three different household members watch three different live channels. Because concurrency cap (4) > upstream cap (2), all three get grants, but only two ffmpeg sessions can exist; the third's spawn kills one of the first two. Each evicted viewer's player re-polls, respawns, and evicts another — all three see intermittent buffering/black-screen stalls and generic 503s, and none is ever told 'provider connection limit reached'. The 429 concurrency signal that WOULD explain it never fires because the concurrency cap is set higher than the real upstream ceiling.
- **Fix:** Either (a) clamp IPTV_MAX_CONCURRENT_STREAMS to IPTV_MAX_UPSTREAM_CONNECTIONS for remux grants so the 3rd viewer gets the structured 429 iptv_concurrency_limit at grant time instead of a silent mid-stream eviction, or (b) make the 'upstream-cap' eviction release the evicted session's concurrency slot AND surface a terminal reason its client can render. Document that the two caps must satisfy CONCURRENT ≤ UPSTREAM for the remux path.
- **Gate:** Test: set concurrency cap 4, upstream cap 2; acquire+spawn 3 remux sessions for 3 subs; assert either the 3rd grant returns 429 iptv_concurrency_limit, or the evicted session is both stopped AND its concurrency slot released with a surfaced reason. Today the 3rd spawns and silently evicts with the slot still held.
- **Verifier note:** value: medium -> low (latent defaults/regression-risk footgun — currently inert in production because ops explicitly pins IPTV_MAX_CONCURRENT_STREAMS=2, matching the upstream default of 2; not an active user-facing incident today). Recommend the fix still land (a startup assertion that CONCURRENT<=UPSTREAM, or wiring eviction to release the slot + surface a reason) purely to prevent this from silently re-igniting on a future config change, not as a hotfix for an ongoing outage.
