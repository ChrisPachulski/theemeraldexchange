# Product Vision Dossier — The Emerald Exchange

---

## 1. WHAT

The Emerald Exchange is an invite-only, self-hosted streaming platform — think of it as a private version of Netflix that runs on a computer you own at home (called a NAS, or Network Attached Storage). The owner of the system installs it, invites trusted people, and everyone gets a polished app to browse a shared media library, watch live TV channels, and request new movies or shows. There is no public sign-up; you are either on the invite list or you cannot get in. The web interface works today in any browser, and the long-term goal is beautiful native iOS and tvOS apps good enough to ship on Apple's App Store. Four programs work together to make it run: a React website the user sees, a TypeScript backend that handles requests and authorization, two Rust services that manage the media library and transcode video for different devices, and a Python recommender that figures out what to suggest next. All of that complexity is invisible to the person watching — they just see a quiet, dark, emerald-tinted interface and their show starts playing.

## 2. WHY

### Why build it instead of using Plex or Netflix?

**Plex crossed a line in 2026.** It now paywalls remote streaming of your own media (files you own, on a server you bought), raised its lifetime price from $250 to $749.99, and keeps adding marketing tiles for content you do not own. You are renting access to your own files. Netflix, Disney+, and their peers own the content and can remove it; you own nothing.

**Jellyfin (the free alternative) has broken native Apple clients.** Swiftfin tvOS v1.0.1 is completely broken on tvOS 26. The iOS app lacks Picture-in-Picture and mid-playback subtitle switching. Free is not good enough if the app does not work.

**The wedge the product exploits:** self-hosting is growing, and a wave of Plex refugees was expected around July 1, 2026 (Plex's price-hike date). There is genuine market space for a privacy-respecting self-hosted server with a genuinely excellent native iOS/tvOS client that never paywalls streaming your own media.

### Why invite-only instead of public sign-up?

Because the app runs on a server you own in your home. You are not operating a public service; you are sharing your personal library with specific people you trust. Public sign-up would mean strangers consuming your bandwidth and hardware. The invite-only constraint is not a limitation — it is the point. It also simplifies security dramatically: instead of managing thousands of unknown users, the system has to reason about a small, vetted list.

### Why four separate programs instead of one?

Each program is in the language and runtime best suited to its job, and each is isolated so a crash or security failure in one does not bring down the others:

- **React/TypeScript (browser):** the user-facing web interface runs in the user's browser. It is fast to build, easy to iterate on, and cannot touch the server's filesystem or database directly — that is a security boundary, not just a convenience.
- **Hono/TypeScript (backend):** JavaScript is the ecosystem's lingua franca for API servers. Auth integrations, database access, and bridging to external services (Sonarr, Radarr, SABnzbd, the recommender) all live here.
- **Rust (media-core, transcoder):** media scanning involves parsing thousands of files and doing math-heavy video transcoding. Rust is the right tool when you need to be fast, safe with memory, and close to the OS (the Intel GPU hardware interface requires calling low-level driver APIs). A bug in transcoding should never take down authentication.
- **Python (recommender):** the data-science and ML ecosystem lives in Python. The recommender uses libraries (sqlite-vec, scipy) with no meaningful Rust or TypeScript equivalents.

### Why the trust boundary is the keystone concept

Every security decision in the codebase traces back to one line: the user's device is untrusted; the NAS is trusted. A user's browser can be compromised, inspected, or spoofed. The NAS is behind a firewall you control. This means:

- The browser never gets direct database access — all requests go through the backend API, which enforces the invite list.
- Video stream tokens are short-lived HMAC credentials generated by the backend, never the browser. The browser cannot forge a token to access a file it was not granted.
- Internal services (media-core, transcoder) authenticate to each other using short-lived encrypted JWE tokens minted by the backend. Even if an attacker got inside the Docker network, they could not impersonate the backend without the secret key.
- The recommender's household signals (what you liked, what you hated) never leave the NAS. They never go to an external API.

### Why the design is as constrained as it is

The product is used nightly by people who want to find a show and watch it in under 30 seconds. Anti-patterns it explicitly refuses: marketing tiles for content you do not own (Plex), admin-complexity bleeding into the consumption surface (Sonarr/Radarr), identical tile-grid sameness (Homarr). The design rules (dark-only, locked OKLCH palette, no gradient text, no side-stripe borders, pill nav) exist so that every implementation decision serves someone relaxing on a couch at 9:42pm, not a developer showing off their design system.

### Why ship to the App Store instead of staying a homelab project

The App Store is the only credible distribution path for iOS/tvOS apps. A homelab web app stays a personal project; a polished native app on the App Store can reach thousands of people who want exactly this but cannot build it themselves. That audience is real, and the Plex refugee timing is a gift. Monetization is planned as: core self-hosting is free forever (remote streaming of your own media is never paywalled), a paid "Pass" unlocks power-user features (hardware transcoding, concurrent streams, multi-user), sold on a web storefront (Stripe) to avoid Apple's 15-30% commission cut, with the iOS app free and purchase-silent.

### How the product vision maps to the four runtimes

```
User's Device (UNTRUSTED)
  |
  +-- React SPA (src/)
        Renders UI, handles user interactions
        Talks ONLY to /api/* via HTTP
        Never touches DB or files directly
        Served from Netlify (a public CDN)
        |
        v (HTTPS via Cloudflare Tunnel)

NAS (TRUSTED — your hardware)
  |
  +-- Hono/TS Backend (server/)
  |     Auth (Plex OAuth / SIWA / WebAuthn passkeys)
  |     Invite/members allowlist enforcement
  |     SQLite data layer
  |     Bridges to Sonarr, Radarr, SABnzbd
  |     Mints internal JWE tokens for Rust services
  |     |
  |     v (internal Docker network, JWE-authenticated)
  |
  +-- Rust media-core (crates/media-core/)
  |     Scans NAS media library (793 movies, 292 shows)
  |     Fetches TMDB metadata
  |     Decides: direct-play vs transcode-required
  |     Serves HLS manifests + stream grants
  |     |
  |     v (internal Docker network, JWE-authenticated)
  |
  +-- Rust transcoder (crates/transcoder/)
  |     ffmpeg sessions: HEVC -> H.264, HDR -> SDR, etc.
  |     Hardware-encode via Intel VAAPI on NAS iGPU
  |     Session concurrency caps (no overwhelming the box)
  |
  +-- Python recommender (recommender/)
        Household signals (likes/dislikes) stored locally
        sqlite-vec scoring; never leaves NAS
        Co-engagement retrieval + cast/crew fusion
```

The Cloudflare Tunnel is the only inbound path from the internet to the NAS. The NAS never opens a port on the home router; `cloudflared` dials out and holds the tunnel open. If the tunnel dies, the site is unreachable even if every other service is healthy.

## 4. PREREQUISITES

A true beginner working through this codebase should have the following concepts in place first. Each is listed with the one reason it is load-bearing for understanding this product specifically.

| Concept | Why you need it here |
|---|---|
| **Client/server model** (browser makes requests, server responds) | Every feature works this way; without this, the four-runtime architecture is just four mystery boxes |
| **HTTP basics** (GET/POST, status codes 200/401/403/503, headers) | Auth failures, stream grants, and error toasts are all HTTP status codes; reading them is how you debug |
| **What a REST API is** | `/api/*` is the only channel between the untrusted browser and the trusted NAS; understanding what the API is explains the trust boundary |
| **JSON** | All data crossing the trust boundary is JSON; every API contract, internal claim, and stream token is serialized JSON |
| **What a database is** (rows, tables, queries) | Auth, invites, watch state, EPG, and feedback all live in SQLite; without this, "migrations" and "schema" are noise |
| **What Docker is** (containers as isolated processes) | The NAS runs 9 Docker containers; deploy means "update a container image"; understanding isolation explains why the transcoder crashing does not kill auth |
| **DNS basics** (what a hostname resolves to) | The Cloudflare Tunnel is the inbound path; "the site is down" almost always means the tunnel or DNS, not the app |
| **What a cryptographic token is** (signed data that cannot be forged) | Stream tokens, device tokens, and internal JWE claims are all cryptographic tokens; the entire security model rests on "only the server can mint these" |
| **What TypeScript is** (JavaScript + types) | Backend and frontend are both TypeScript; types are how the compiler catches mistakes before they reach production |
| **What Rust is** (compiled, memory-safe system language) | Two of the four runtimes are Rust; knowing "Rust is compiled, fast, and crashes rather than has memory corruption" explains why media and crypto live there |
| **What Python is** (scripting language, ML ecosystem) | The recommender is Python; knowing "the data-science libraries exist in Python" explains why the fourth runtime is not TypeScript |

## 5. GOTCHAS AND WAR STORIES

These are real lessons documented in the project, relevant to the product-vision layer.

**"The Watch button is a deliberate cheat."**
PRODUCT.md records a formal exception: the `Watch` nav entry currently opens Plex's hosted web client (app.plex.tv) in a new tab, and the per-title `DetailModal` has a "Play in Plex" link. These exist because native in-browser playback (via media-core + the transcoder) has shipped but has not yet reached parity with Plex's own client — resume everywhere, subtitles, downloads visibility are still missing. The strategic principle says "the app is the experience, no links to underlying services" — but the principle records its own exceptions explicitly, with a bounded exit condition: both links are removed when native playback reaches parity. The lesson: a strong design principle is not violated by documenting a time-bounded exception; it is violated by pretending the exception does not exist.

**"The invite list IS the product."**
There is no admin/family split in the UI. Household members and invited remote members get the same interface. Owner-only administration (invites, member management, device management) is gated by the backend, not hidden by a client-side toggle. The lesson: conflating "who sees the feature" (a client-side secret mode) with "who is allowed to use the feature" (a server-side check) is a security mistake. The backend enforces; the frontend reflects.

**"Local-first personalization was not just about privacy — it was about cost and trust."**
The recommender runs on the NAS so household signals never leave the network. This is positioned as both a privacy feature (your viewing habits go nowhere) and an independence feature (the product does not depend on an external ML API that could go down, change pricing, or be acquired). The lesson: "local-first" means you own the data AND the infrastructure, and those are two separate benefits that both matter.

**"The Apple gate is the dominant constraint on the roadmap."**
M2 (Swift SDK + iOS/tvOS apps), M5 (native media clients + offline downloads), and M6 (monetization) are all blocked on Xcode being installed and the Apple Developer Program account being activated. Zero Swift lines exist. The backend substrate they would consume is built and proven. The lesson: hardware-side readiness (NAS is humming, API is live) and software-gate readiness (Xcode, developer account) are independent, and the latter blocked six months of planned work entirely.

**"ffmpeg GPL contamination is the single hardest blocker to shipping commercially."**
The backend and media-core images bundle a statically-compiled GPL ffmpeg binary. GPL-3.0 is incompatible with the App Store (Apple's license terms contain restrictions GPLv3 forbids). The good news: the Rust transcoder calls ffmpeg as a separate process (no linking), so the contamination is only the bundled binary, not the source code. The fix is surgical — rebuild ffmpeg LGPL-only for the iOS bundle, use VideoToolbox (Apple's native hardware codec). The lesson: an otherwise-clean codebase can have a single dependency that blocks distribution entirely; license auditing is not a formality.

**"The Plex refugee timing is a real strategic window, not a marketing story."**
Plex announced a price hike from $250 to $749.99 lifetime on July 1, 2026, AND paywalled remote streaming of your own media. Jellyfin's Apple TV client is completely broken on the latest tvOS. A polished, privacy-first, free-to-stream-your-own-media native iOS/tvOS app launched before July 2026 catches a real wave of people actively looking for an alternative. The lesson: timing matters in product work; the market window is documented in the codebase as a constraint, not a hope.

## 6. QUIZ BANK

**Q1. The owner decides to show users a "Settings" button that links directly to the Sonarr configuration panel. What PRODUCT.md principle does this violate, and what is the correct approach?**

Answer: It violates strategic principle 1 ("The app is the experience — no links to the underlying services inside it"). Sonarr is implementation detail; members should never need to know it exists. The correct approach is to surface whatever configuration capability is needed as a native feature within the app itself. If the capability is genuinely owner-only (operator tooling), PRODUCT.md records one explicit exception: admin-only app links (Sonarr/Radarr/SAB) are permitted in the user menu, gated strictly on `isAdmin` so members never see them. The lesson is that there IS a permitted path — but it is owner-gated, not member-facing.

**Q2. A new engineer proposes storing recommendation signals (likes/dislikes) in a cloud database so the recommender can run faster machines. What breaks, and why does PRODUCT.md prohibit this?**

Answer: It breaks strategic principle 5 ("Local-first personalization — member signals never leave the NAS") and destroys a core product promise. The product is positioned as privacy-respecting and self-hosted; shipping your viewing behavior to an external service contradicts that entirely, makes the product dependent on an external service that could go down or raise prices, and creates GDPR/CCPA disclosure obligations. The recommender is explicitly a sidecar on the NAS for both privacy and cost reasons.

**Q3. Milestone M5 (native iOS/tvOS media clients) is 3% complete. The backend media-core is live and proven. Why can M5 not be built despite the backend being ready?**

Answer: M5 requires Xcode to be installed and an active Apple Developer Program account to sign and deploy any Swift code. Without those, no Swift line can compile, no AVPlayer code can run on a real device, no TestFlight build can be created. The backend readiness and the Apple tooling gate are completely independent. The dependency is one-directional: M5 needs the Apple gate plus the backend (both), and the Apple gate has nothing to do with whether the Rust API is working.

**Q4. Why does cloudflared being restarted after any backend container recreate matter to a user trying to watch a movie?**

Answer: cloudflared shares the backend container's network namespace (`service:backend` in docker-compose). When the backend container is recreated, its network namespace is destroyed and replaced; cloudflared's reference to the old namespace breaks, so it can no longer route traffic to the backend. The Cloudflare Tunnel is the only inbound path from the internet to the NAS — there is no router port forward. If cloudflared cannot reach the backend, the entire public site returns 502 or 530 even though every other service is healthy. A user trying to watch gets a dead page, not a graceful error.

**Q5. The monetization plan says "never paywall streaming your own media." Why is this a strategic principle and not just a nice-sounding promise?**

Answer: It is the exact line Plex crossed in 2025-2026 (remote streaming of your own files became paywalled), which is the primary reason the Plex-refugee market exists. The product's positioning is "the privacy-respecting self-hosted server that never does what Plex did." Paywalling remote streaming would erase the entire differentiation and send the target audience back to Jellyfin (free) or a resigned Plex subscription. The principle is documented explicitly so no future entitlement design accidentally gates it — MONETIZATION.md's feature table marks it "Free: always" with the explicit note "Do NOT gate: basic local playback or remote streaming of your own media. That is the line Jellyfin gives away free and Plex crossed."

**Q6. Why can the iOS app not simply bundle the same ffmpeg binary the NAS transcoder uses?**

Answer: The transcoder's Debian-packaged ffmpeg is compiled with GPL-3.0+ components (libx264). GPL-3.0 is incompatible with the App Store because Apple's Developer Program License Agreement imposes use and distribution restrictions that GPLv3 section 7 forbids. Apps containing GPL-3.0 code cannot be distributed through the App Store. The NAS server can distribute GPL ffmpeg separately (with source offer and license notices) because it is a server the owner installs on their own hardware — that is a different legal context than an Apple-distributed binary. The correct iOS path is to use VideoToolbox, Apple's native hardware decode/encode API, which has no GPL contamination.
