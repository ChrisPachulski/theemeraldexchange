### Plan 002 deviation (orchestrator decision)

The plan's premise — add a `start_utc`-leading index and the planner uses it —
proved FALSE for this codebase (verified empirically at 600k rows):

- A new index is NOT chosen without statistics; SQLite full-SCANs regardless.
- Once stats exist, the planner prefers the EXISTING `(channel_id, start_utc)`
  PK index (skip-scan), which ALSO serves `ORDER BY channel_id` for free — a new
  2-col index would force a sort and is never selected.

So the real lever is **statistics, not a new index**. Implemented fix: `syncOnce()`
runs `PRAGMA optimize` after repopulating `epg_programs` (~15ms at 600k rows,
best-effort), flipping the guide query from `SCAN` to `SEARCH`. No migration 0007.
Regression test pins the invariant (`SCAN` before stats, `SEARCH` after optimize).
Touched `iptvSync.ts` (plan marked it out-of-scope) — justified amendment.
