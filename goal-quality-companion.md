Source: the same 150-agent adversarially-verified audit as the Apple client's
`~/theemeraldexchange-apple/goal-quality.md` (115 raw → 111 survived + 13 critic adds,
deduped to 111 items across the two docs). This doc is the 38-item server half. Every item
carries verifier-checked citations, a concrete failure scenario, and an objective red→green
gate; ADJUSTED items carry the verifier's corrections inline.

Verification gates for this repo: `npm test` (vitest route/contract suites), `cargo test`
+ `cargo clippy` for the crates, and the Playwright suite where a web-UI item says so.
NAS deploys ONLY via nas-safe-build discipline — never a raw compose build on the box
(Tier S0 #1 removes the remaining footgun).

Priority order: S0 (can take down Plex / lose member data) → S1 (the multi-audio flagship
is currently dead-on-arrival, #10 there) → S2 (policy bypasses) → S3–S7. Within a tier,
order is severity-sorted.

### 6. Trickplay default-ON (S5) spawns an uncapped, unmetered full-source software decode per re-encode start on the 6-thread NAS — the exact Plex brown-out failure mode the module doc itself warns about

- **Where:** `/Users/cujo253/theemeraldexchange/crates/transcoder/src/trickplay.rs:86-107 (default ON when env unset), :24-26 (doc admits 'the fps sampler still walks every frame' on 'the repo's most-warned-about failure mode' box), :333-371 (thumb_args: no -threads cap, no -skip_frame nokey, no -readrate, no hwaccel — full-rate software decode of the whole source); /Users/cujo253/theemeraldexchange/crates/transcoder/src/session.rs:1086-1088 (spawned on every re-encode start), :1041-1046 (only the MAIN session charges the Limiter); /Users/cujo253/theemeraldexchange/crates/transcoder/src/concurrency.rs:1-10 (caps count sessions only); docker-compose.yml transcoder service (~line 443-465): no TRANSCODER_TRICKPLAY set and no container cpus limit, so the next deploy ships default-ON.`  (S · high value)
- **Failure:** User plays a 4K HEVC title that re-encodes (e.g. HDR tonemap to h264). The main encode may ride VAAPI, but the thumbnail pass software-decodes the entire 4K source at maximum speed on all 6 threads (fps=1/10 still decodes every frame; libx264/decode default -threads = all cores) for potentially the whole feature length on this CPU — while charging ZERO concurrency slots. Two household re-encode plays = two uncapped full decodes alongside Plex; load spikes like the historical 73-load brown-outs. - **Gate:** Unit (red today): assert thumb_args contains -skip_frame nokey and a -threads cap. ### 36. A backend restart or deploy mid-recording silently finalizes a partial DVR recording as 'completed' and the scheduler can never resume it — data loss labeled as success
